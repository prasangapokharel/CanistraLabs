KEY FINDINGS
================================================================================

STATISTICS:
- 55 API endpoints across 8 routers
- 12 core service modules (3,761 LOC)
- 8 data models + 2 token models
- 10 database tables with CASCADE relationships
- 2 token types (access + refresh, 24h + 7d)
- 4 deployment statuses + 5 project statuses
- 7 domain verification states

DATABASE ARCHITECTURE:
- PostgreSQL with SQLAlchemy ORM
- User is the central entity (1:N to Projects, Enrollments, Domains)
- Projects relate to: Deployments, Canister, Metrics, CustomDomains
- All relationships use CASCADE on delete
- Unique constraints: email, username, principal_id, account_id, canister_id, url
- Indexed fields: user_id, project_id, canister_id, principal_id, domain_name

AUTHENTICATION:
- JWT-based (access + refresh tokens)
- Algorithm: HS256
- Access token: 24 hours validity
- Refresh token: 7 days validity
- Token storage: localStorage in browser
- Transmission: Authorization: Bearer {token} header
- Password hashing: Argon2
- ICP integration: Each user gets unique principal_id + account_id (encrypted)

SERVICE LAYER (12 services):
1. AuthService - User registration, login
2. ProjectService - CRUD, URL generation
3. DeploymentService - Deployment orchestration
4. ICPIdentityManager - ICP identity lifecycle
5. CanisterFactory - Per-project canister creation
6. DomainManagementService - Custom domains, DNS, SSL
7. DfxCommand - Low-level ICP CLI wrapper
8. DynamicDeploymentService - Dynamic deployment handling
9. AutoFundingDetector - Wallet balance checking
10. RosettaClient - ICP Ledger API integration
11. EmailVerificationService - Email token lifecycle
12. PasswordResetService - Password reset workflow

API ENDPOINTS (55 total):
- Authentication (8): signup, login, refresh, logout, me, forgot-password, reset-password, verify-email
- Projects (5): CREATE, READ (list), READ (single), UPDATE, DELETE
- Deployments (7): deploy, status, history
- Wallet (12): identity, balance, transfer, convert, funding, etc.
- Domains (8): setup, verify, status, register, delete, list
- Metrics (4): project metrics, live metrics, dashboard
- Canister (5): create, update, status, delete, DNS help
- Cron (3+): start, stop, trigger, status

CRITICAL BUSINESS LOGIC:
1. Deterministic project URL generation (md5 hash + project_id)
2. Automatic ICP identity creation on user signup
3. Per-user encrypted identity storage (AES-256)
4. Per-project canister creation with unique controller
5. Custom domain setup with multi-step DNS verification
6. Wallet balance tracking from ICP Ledger (Rosetta API)
7. Automatic cycles conversion from ICP
8. RBAC with 4 roles: owner, admin, editor, viewer

MOTOKO MIGRATION STRATEGY
================================================================================

PHASE 1 (Weeks 1-2): Core Infrastructure
- Deploy 7 empty canisters (allocate cycles)
- Implement Users canister (registration, profiles)
- Implement Projects canister (CRUD, ownership)
- Set up inter-canister communication patterns
- Add retry/resilience logic

PHASE 2 (Weeks 3-4): Deployment & Infrastructure
- Implement Deployments canister
- Integrate Management canister API
- Test canister creation and code installation
- Implement status tracking

PHASE 3 (Weeks 5-6): Advanced Features
- Implement Domains canister
- Implement Wallet canister
- Implement Metrics canister
- HTTP outcalls for DNS & boundary nodes

PHASE 4 (Week 7): Testing & Migration
- End-to-end testing
- Data migration scripts
- Parallel running (write to both systems)
- DNS cutover

PHASE 5 (Week 8+): Cutover & Monitoring
- Switch to Motoko backend
- Monitor for issues
- Sunset Python backend (keep for 30 days)

RECOMMENDED CANISTER DECOMPOSITION:
1. Users Canister (~100KB per 1000 users)
   - Registration, profiles, authentication state

2. Projects Canister (~1MB per 10,000 projects)
   - CRUD operations, ownership, enrollments

3. Deployments Canister (~500KB per 1000 projects)
   - Canister creation, code deployment, lifecycle

4. Domains Canister (~200KB per 1000 domains)
   - Custom domain config, DNS verification, SSL

5. Wallet Canister (~50KB per 1000 users)
   - Balance tracking, transfers, conversion logic

6. Metrics Canister (1MB+ unbounded)
   - Time-series metrics storage and retrieval

7. Authorization Canister (Optional, ~100KB per 10K enrollments)
   - Centralized RBAC verification

KEY DESIGN DECISIONS:
- One canister per domain (isolation, independent upgrades)
- No shared state between canisters (reduces deadlocks)
- Async all the way (Motoko default)
- Stable memory for state persistence
- Error codes via Result types (no exceptions)
- No JWT in canister (validate at API gateway, use principal inside)
- Reverse indices with HashMap (no DB unique constraints in Motoko)
- Bounded collections with BTreeMap (prevent OOM)

DATA MIGRATION APPROACH:
1. Deploy empty Motoko canisters
2. Snapshot Python PostgreSQL database
3. Transform and batch load into Motoko canisters
4. Verification phase (compare row counts, hashes)
5. Parallel running period (1-2 weeks)
6. DNS cutover to Motoko canisters
7. Sunset Python backend after 30 days

KEY CHALLENGES FOR MOTOKO:
- No UNIQUE constraints → Need reverse index HashMaps
- Metrics table grows unbounded → Use circular buffer
- HTTP outcalls needed for DNS and boundary nodes
- Inter-canister calls need retry logic and circuit breakers
- Token validation moves to API gateway
- Large data migration from PostgreSQL

AUTHENTICATION & AUTHORIZATION
================================================================================

JWT FLOW:
1. User signs up/logs in
2. AuthService validates credentials (email + password)
3. Create JWT tokens:
   - Access token (24 hours, HS256)
   - Refresh token (7 days, HS256)
4. Frontend stores in localStorage
5. Every request includes Authorization: Bearer {token}
6. FastAPI middleware validates token, extracts user_id
7. Each endpoint checks ownership/permissions

OWNERSHIP VERIFICATION:
- All endpoints use: ProjectService.get_project_by_id(session, project_id, user_id)
- Returns None if user doesn't own project or project doesn't exist
- Implicitly prevents access without exposing project existence

RBAC (Role-Based Access Control):
- 4 roles: owner, admin, editor, viewer
- Stored in ProjectEnrollment table
- Not yet enforced in current implementation (binary ownership only)
- Ready for enhancement

ICP PRINCIPAL INTEGRATION:
- Each user gets: principal_id + account_id (created at signup)
- ICP identity keys encrypted with AES-256 and stored
- Used for: Canister creation, deployment, cycles management
- Flow: User → Principal ID → Canister Controller → Deployed App

================================================================================
CRITICAL INTEGRATION POINTS
================================================================================

1. User Registration → ICP Identity Creation
   - Automatic on signup via ICPIdentityManager
   - Creates dfx identity with unique name
   - Exports and encrypts private key
   - Stores principal_id + account_id for future deployments

2. Project Deployment → Canister Creation
   - User identity restored from encrypted storage
   - DFX context switched to user's identity
   - New canister created with user as controller
   - Code installed and URL generated

3. Custom Domain Setup → DNS Verification → SSL Issuance
   - Domain config created
   - DNS instructions generated
   - User configures DNS at registrar
   - System verifies DNS records
   - ICP boundary nodes register domain
   - SSL certificate auto-issued via ACME

4. Wallet Management → Rosetta API → Cycle Conversion
   - Check ICP balance via Rosetta API
   - Check cycles balance via Management Canister
   - Determine if funding required
   - Auto-conversion available if ICP > 0

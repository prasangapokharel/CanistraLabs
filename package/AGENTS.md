# Repository Instructions

Use these instructions for everything in this workspace unless a more specific file overrides them.

## General Rules

1. Keep changes small, direct, and easy to review.
2. Prefer existing repo patterns over inventing new ones.
3. Use official documentation when behavior is unclear.
4. Do not create new Markdown files unless the user explicitly asks for documentation.

## Python Rules

1. Keep Python code minimal and readable.
2. Prefer small functions and short files when practical.
3. Use `camelCase` for new Python identifiers, including variables, functions, and file-local helpers, unless you must match an existing API or language requirement.
4. Avoid docstrings unless they add real value.
5. Use brief comments only when the code would otherwise be hard to understand.

## Naming Rules

1. Use `camelCase` for new identifiers by default.
2. Keep naming consistent within a file and across related files.
3. Match existing names when editing existing code instead of renaming for style alone.

## Markdown Rule

- Never create new `.md` files unless the user explicitly requests them.
